Bottom line: separate restore points from offsite retention

Backup design

Daily backups are only one layer

Kinsta documents daily, manual, system-generated, hourly, external, and downloadable backups. Combine pre-change restore points with offsite or local retention based on the site’s recovery objective.

What a restore changes

Kinsta describes its snapshots as including files, database, redirects, Nginx configuration, MyKinsta settings, and domain state. Changes made after the selected restore point can be lost, so capture the current state before restoration.

Safer workflow: restore to staging first when possible, then validate rendering, forms, payments, and admin functions.

Standard change workflow

  1. 1.Create and label a manual backup before the change.
  2. 2.Test plugin, theme, and PHP changes in staging.
  3. 3.After production deployment, test forms, payments, cron, and cache.
  4. 4.Before restoring, identify data created after the restore point.
  5. 5.Use external or downloadable copies for critical retention.

Backup frequency by site type

Low-change sites

  • Daily plus manual pre-change backup
  • Periodic downloadable archive
  • Documented restore test

Transactional sites

  • Define acceptable data-loss window
  • Consider six-hour or hourly add-ons
  • Document offsite and restore procedures

FAQ

How many manual backups can Kinsta store?

The documentation states up to five manual backups.

Can I download a backup?

A downloadable backup can be generated once per week and remains available for 48 hours.

Can a backup be restored to staging?

Kinsta documents restoring a backup to a staging environment.
